The Problem

Professionals with non-linear backgrounds — people who've combined design with finance, or technical work with product strategy — struggle to articulate where they fit in the job market.

Traditional career tools assume you fit a clean archetype. They don't account for the reality that many valuable professionals span multiple domains and their strongest matches aren't obvious from a job title alone.

The result: misapplied CVs, wasted applications, and a persistent feeling that "I'm qualified but I don't know what to call myself."

The Solution

TryCareerMatch asks users to rate their proficiency across 88 skills — from financial modelling and SQL to stakeholder management and UX thinking. A weighted scoring model then calculates their fit against 60+ roles in 9 sectors.

The output isn't a single "best match" — it's a ranked view of roles where the user's actual skill profile produces the strongest alignment, including surprising secondary matches they may never have considered.

No account, no paywall, no email capture. Just results.

Process

How It Was Built

From the initial idea to a working product, in roughly 6 weeks.

01
Research & Scoping
Mapped existing career tools, identified the gap for cross-functional profiles. Defined the scope: which skills, which roles, which sectors would produce meaningful signal.
02
Data Model Design
Designed the 88-skill taxonomy across 8 categories. For each role, assigned per-skill importance weights (0–3). The scoring logic: user proficiency × role weight, normalised to a 0–100 match score.
03
UX & Interface
Designed the skill assessment UI — quick, no-friction sliders with clear labels. Results screen showing role match %, sector breakdown, and top-skill alignment highlights.
04
Build & Launch
Built entirely in vanilla HTML/CSS/JS — no framework, no database. All logic runs client-side. Deployed via Vercel. Promoted through the portfolio's announcement bar.
Under the hood

The Scoring Model

Each of the 60+ roles has a skill importance profile — a vector of weights across the 88 skills, from 0 (irrelevant) to 3 (critical).

When a user rates their skills, the model computes a dot product between their proficiency ratings and each role's importance weights, then normalises the result to a 0–100 match percentage.

This means a Data Analyst with strong SQL and Power BI but no stakeholder management experience will score higher on technical analyst roles than on product manager roles — even if both appear in their results.

The skill taxonomy spans 8 categories: Business Intelligence, Data Engineering, Financial Analysis, Product & Strategy, Business Analysis, Design & Creative, AI & Tools, and Languages.

Technology

Stack

Deliberately minimal — the complexity is in the data model, not the framework.

HTML / CSS / JS
Entirely vanilla — no framework. All scoring logic runs client-side with no backend required. Fast and zero infrastructure cost.
Weighted Scoring Model
Custom-built data model: 88-skill taxonomy × 60+ role importance matrices. Dot product scoring normalised to 0–100%. Designed from scratch.
Figma (UX Design)
Full UI design and prototyping in Figma before implementation. Designed for clarity and speed — users should reach results in under 3 minutes.
Vercel (Hosting)
Deployed on Vercel for zero-configuration HTTPS, global CDN, and automatic deploys from Git. No backend means no server costs.
Git / GitHub
Version control and deployment pipeline via GitHub. Feature branches for major scoring model changes, with tagged releases.
AI-Augmented
Claude used throughout — for generating role skill matrices, refining UX copy, debugging edge cases in the scoring logic, and accelerating CSS work.
Retrospective

Lessons Learned

What worked, and what I'd approach differently next time.

What worked
Staying vanilla
No framework meant no build tooling, no dependency management, and zero deployment friction. For a project of this complexity, vanilla JS was the right call — fast to iterate and easy to debug.
What worked
Designing the data first
Spending the first two weeks entirely on the skill taxonomy and role matrices — before writing a single line of UI — paid off enormously. The UX design was straightforward once the data model was solid.
What worked
No-registration friction
Removing all friction from the experience — no account, no email, immediate results — was the right product decision. The conversion rate from landing to completing the assessment is high.
What I'd change
More granular skill weighting
The current model uses integer weights (0–3). A continuous scale or sub-skill breakdowns would produce more nuanced results. I'd also add a "seniority" dimension — the importance profile for a junior vs. senior role differs significantly.
What I'd change
Add a saved results flow
Users currently can't save or revisit their results without re-doing the assessment. A lightweight localStorage-based "save session" feature or a shareable results URL would significantly improve retention and word-of-mouth sharing.
What I'd change
Analytics from day one
I added analytics after launch. Starting with even basic event tracking (which skills users score highest, where they drop off in the assessment) would have given me much better data to improve the product in week 2.
